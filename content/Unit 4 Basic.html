Slide 2: Introduction to Bitcoin and Blockchain

  • What is Bitcoin?
    • First decentralized cryptocurrency (launched in 2009)
    • Created by Satoshi Nakamoto
  • What is Blockchain?
    • Distributed ledger technology (DLT)
    • Stores transactions in blocks linked by cryptography

Slide 3: How Bitcoin Blockchain Works

  • Transactions are verified by nodes
  • Grouped into blocks
  • Blocks linked together using cryptographic hash functions
  • Decentralized and immutable

Let’s break down the key components and how they work together:

🔹 1. Transactions

A Bitcoin transaction is a transfer of value between two parties on the Bitcoin network.

  • Transactions involve inputs and outputs:
    • Inputs – The source of Bitcoin (linked to a previous transaction)
    • Outputs – The recipient’s address and the amount being sent

➡️ Example of a Simple Transaction:

  • Alice wants to send 0.5 BTC to Bob.
  • Alice has an unspent output (UTXO) of 1 BTC.
  • Alice creates a transaction with:
    • Input = 1 BTC (linked to a previous UTXO)
    • Output 1 = 0.5 BTC to Bob’s address
    • Output 2 = 0.49 BTC back to Alice (change)
    • Transaction fee = 0.01 BTC

🔹 2. Digital Signatures and Public Key Cryptography

Bitcoin uses Elliptic Curve Digital Signature Algorithm (ECDSA) for transaction authentication.

  • Every Bitcoin wallet has a private key and a public key:
    • Private Key – Secret key known only to the wallet holder (used to sign transactions)
    • Public Key – Used to generate the Bitcoin address (shared openly)
  • To authorize a transaction:
    • Alice signs the transaction with her private key.
    • Nodes verify the signature using Alice’s public key.

✅ Why It’s Secure:

  • ECDSA ensures that only the owner of the private key can authorize a transaction.
  • Even if someone knows the public key, they cannot reverse-engineer the private key.

🔹 3. Transaction Propagation and Validation

Once Alice signs the transaction:

  1. The transaction is broadcast to the Bitcoin network (peer-to-peer).
  2. Nodes (computers in the network) validate the transaction by checking:
    ✅ Inputs are not already spent (prevents double-spending)
    ✅ Signature is correct (proves authenticity)
    ✅ Transaction follows consensus rules (e.g., fees, size limits)

✅ How Double Spending Is Prevented:

  • Each input can only be spent once.
  • The first valid transaction propagated to the network gets confirmed in a block.

🔹 4. Block Formation

Once transactions are validated, they are grouped into a block:

  1. Miners collect unconfirmed transactions from the mempool (waiting area).
  2. They create a block header that includes:
    • Previous block hash (links the block to the previous one)
    • Merkle root (hash of all transactions in the block)
    • Timestamp
    • Nonce (random value used for mining)
  3. Miners compete to find a valid hash using Proof of Work (PoW):
    • Hash must meet the network’s current difficulty target.
    • Requires massive computational power (brute force).

✅ Merkle Tree Structure:

  • Transactions are organized into a Merkle Tree.
  • The Merkle Root summarizes all transactions.
  • If even one transaction changes, the Merkle Root changes—ensuring integrity.

Example of a Merkle Tree:

            Merkle Root
               /  \
            /      \
         Hash A   Hash B
         /   \     /   \
      Tx1   Tx2  Tx3   Tx4

🔹 5. Proof of Work (PoW) and Mining

To add a block to the blockchain, miners must solve a cryptographic puzzle:

  1. Find a nonce value that produces a block hash that meets the difficulty target.
  2. Example:
    • Target = 0000000000000xxxxxxxxxxxxxxxx
    • Valid hash starts with a certain number of leading zeros.
  3. When a miner solves the puzzle:
    • The block is broadcast to the network.
    • Other nodes verify the solution.
    • The winning miner receives a block reward + transaction fees.

✅ Mining Rewards:

  • Current block reward = 6.25 BTC (as of 2024)
  • Block reward halves approximately every 4 years (next halving expected in 2028).

🔹 6. Blockchain Linkage and Finalization

  • Each block references the hash of the previous block → Creates a chain of blocks
  • Immutable because:
    • Changing one block would require recalculating the hash of every subsequent block.
    • Computationally impossible due to PoW difficulty.
  • Longest valid chain = the most computational work → Considered the true chain.

Example of Blockchain Structure:

[Block 1001] → [Block 1002] → [Block 1003] → ...
   Hash: abc123     Hash: def456     Hash: ghi789

🔹 7. Confirmation and Finality

  • A transaction is considered confirmed when it is included in a block.
  • More confirmations = higher security:
    ✅ 1 confirmation = included in one block
    ✅ 6 confirmations = strong security against reversal

Example:

  • Alice sends Bob 1 BTC.
  • Bob waits for 6 confirmations before treating it as final (to avoid chain reorganization).

✅ Summary of Process

Step | Description | Purpose

Transaction Creation | Alice signs a transaction with her private key | Ensures authenticity and ownership

Broadcasting | Transaction sent to nodes for validation | Ensures transaction follows rules

Mining and PoW | Miners compete to solve the puzzle | Secures the network and creates scarcity

Block Formation | Valid transactions added to a block | Ensures data integrity and order

Chain Linkage | Each block references the previous block | Creates immutability

Finalization | More confirmations increase security | Prevents double-spending

✅ Why Bitcoin’s Blockchain is Secure:

✔️ Decentralized – No central authority
✔️ Immutable – Changes require massive computational work
✔️ Secure – Cryptography protects transaction authenticity
✔️ Transparent – All transactions are publicly verifiable

Slide 4: Challenges of Bitcoin Blockchain

  1. Scalability – Limited to ~7 transactions per second (TPS)
  2. Energy Consumption – Proof of Work (PoW) requires high computational power
  3. Transaction Speed and Fees – High during network congestion
  4. Security vs. Decentralization – Trade-offs between speed and decentralization

Slide 5: Solutions to Bitcoin Blockchain Challenges

✅ Lightning Network – Off-chain solution for faster transactions
✅ SegWit (Segregated Witness) – Increases block capacity
✅ Sidechains – Parallel chains for faster transaction processing
✅ Taproot Upgrade – Improves privacy and smart contract capability

Slide 5a: Lightning Network

🔹 What is the Lightning Network?

  • Layer 2 solution built on top of the Bitcoin blockchain
  • Designed to enable fast, low-cost transactions by conducting them off-chain

🔹 How It Works:

  1. Opening a Payment Channel – Two parties open a multi-signature channel on the Bitcoin blockchain.
  2. Conducting Transactions – Unlimited transactions can occur within the channel without being recorded on the main blockchain.
  3. Closing the Channel – When the channel closes, the final state is recorded on the Bitcoin blockchain.

🔹 Benefits:

✅ Instant transactions (milliseconds)
✅ Extremely low fees
✅ Scalable for microtransactions

🔹 Example:

  • Alice and Bob open a channel with 1 BTC each.
  • They can send funds back and forth instantly within this channel.
  • When they decide to close the channel, the net result is recorded on the blockchain.

🔹 Challenges:

  • Network liquidity issues
  • Routing problems for multi-hop payments

Slide 5b: SegWit (Segregated Witness)

🔹 What is SegWit?

  • A protocol upgrade implemented in 2017
  • Separates transaction signatures (witness data) from the transaction data
  • Effectively increases the block size limit without directly increasing the block size

🔹 How It Works:

  1. Original Bitcoin block limit: 1 MB
  2. SegWit moves the signature data to a separate structure, freeing up space within the block for more transaction data.
  3. The effective block size increases to ~4 MB (if fully utilized).

🔹 Benefits:

✅ Increased throughput (more transactions per block)
✅ Reduced transaction fees
✅ Fixes transaction malleability (ensures consistent transaction IDs)

🔹 Example:

  • Pre-SegWit: A block might hold 2,000 transactions
  • Post-SegWit: A block can hold up to 8,000 transactions (depending on signature size)

🔹 Challenges:

  • Slow adoption rate among wallets and exchanges
  • Not all nodes and miners upgraded to SegWit initially

Slide 5c: Sidechains

🔹 What are Sidechains?

  • Independent blockchains that run parallel to the Bitcoin blockchain
  • Linked to the main chain through a two-way peg
  • Allows tokens and assets to be transferred between chains

🔹 How It Works:

  1. Users lock BTC on the main chain.
  2. An equivalent amount is minted on the sidechain.
  3. Transactions occur independently on the sidechain.
  4. The final state is transferred back to the main chain when needed.

🔹 Examples:

  • Liquid Network – Sidechain for faster and confidential Bitcoin transactions
  • Rootstock (RSK) – Smart contract functionality on a Bitcoin sidechain

🔹 Benefits:

✅ Faster transactions
✅ Lower fees
✅ Additional functionalities (e.g., smart contracts)

🔹 Challenges:

  • Trust issues with the sidechain operators
  • Security vulnerabilities in the two-way peg mechanism

Slide 5d: Taproot Upgrade

🔹 What is Taproot?

  • Major Bitcoin upgrade activated in November 2021
  • Enhances privacy, scalability, and smart contract flexibility

🔹 How It Works:

  1. Introduces Schnorr signatures – More efficient than ECDSA (Elliptic Curve Digital Signature Algorithm)
  2. Combines multiple signatures into a single one (improves transaction efficiency)
  3. Merkelized Abstract Syntax Tree (MAST):
    • Allows complex smart contracts to appear as simple single-signature transactions on the blockchain.
    • Improves privacy and reduces block size.

🔹 Benefits:

✅ Greater privacy – Multi-signature transactions appear identical to single-signature ones
✅ Lower transaction fees due to smaller data size
✅ Increased smart contract functionality

🔹 Example:

  • Before Taproot: A multisig wallet transaction is identifiable on the blockchain.
  • After Taproot: A multisig wallet transaction looks identical to a single signature transaction.

🔹 Challenges:

  • Slow adoption due to complexity
  • Smart contract development on Bitcoin remains limited compared to Ethereum

Slide 6: Proof of Work (PoW)

  • Definition: Consensus mechanism used in Bitcoin
  • How It Works:
    • Miners solve complex mathematical problems
    • First to solve gets to add a block to the chain
    • Rewards in Bitcoin
  • Pros: High security and decentralization
  • Cons: High energy consumption and slow transaction speed

Slide 7: Proof of Stake (PoS)

  • Definition: Consensus mechanism used in alternatives like Ethereum 2.0
  • How It Works:
    • Validators stake coins instead of mining
    • Validator is chosen to create new blocks based on stake
  • Pros: Energy-efficient, faster transaction processing
  • Cons: Centralization risk (wealthier participants have more power)

Slide 8: Alternatives to Bitcoin’s PoW

🔹 Delegated Proof of Stake (DPoS): Faster consensus using selected delegates (e.g., EOS)
🔹 Proof of Authority (PoA): Consensus through trusted validators (e.g., VeChain)
🔹 Proof of Burn: Destroy coins to gain mining rights (e.g., Slimcoin)
🔹 Hybrid PoW/PoS: Combines elements of both systems (e.g., Decred)

Slide 9: Bitcoin Scripting Language

  • Purpose: Allows simple smart contracts on Bitcoin
  • Structure: Stack-based (uses opcodes)
  • Capabilities:
    ✅ Multisig transactions
    ✅ Time locks
    ✅ Atomic swaps
  • Limitations: Not Turing-complete (intentional for security)

Slide 10: Conclusion

  • Bitcoin’s blockchain is secure but faces scalability and efficiency challenges
  • Alternative consensus mechanisms provide faster and greener solutions
  • Bitcoin scripting enables basic smart contract functionality
  • Future developments (e.g., Taproot) aim to enhance Bitcoin’s flexibility and performance
