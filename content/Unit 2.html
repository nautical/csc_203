Cryptography - Overview

Slide 1 : Overview of Modern Cryptography

Evolution of Cryptography:

  • Originated thousands of years ago.
  • Early cryptographic algorithms included MD5 and DES (now retired).
  • Modern cryptographic algorithms: SHA-3, Argon2, ChaCha20.

Slide 2 : What is Cryptography?

  • Definition of Cryptography: Science of securing and protecting information.
  • Ubiquity in the Digital World: Used in websites, emails, and WiFi connections.
  • Importance for Developers:
    • Need basic understanding of cryptography.
    • Knowledge of crypto algorithms and libraries.
    • Understanding of hashing, symmetric & asymmetric ciphers, encryption schemes.
    • Familiarity with digital signatures, cryptosystems, and underlying algorithms.

Slide 2.1 : Encryption and Keys

  • Cryptography’s Role:

    • Ensures secure storage and transmission of data.
    • Protects data so only intended recipients can access and process it.
    • Involves encryption and decryption using symmetric or asymmetric schemes.
    • Uses keys to transform data from plain to encrypted form and back.

  • Symmetric Encryption:

    • Same key used for both encryption and decryption.
    • Examples: AES, Twofish, ChaCha20.

  • Asymmetric Encryption:

    • Uses a public-key cryptosystem with a key-pair.
    • Public key (encryption) and private key (decryption).
    • Examples: RSA, ECC.

  • Encryption Schemes:

    • Often combine multiple algorithms.
    • Examples: AES-256-CTR-HMAC-SHA-256, ChaCha20-Poly1305, ECIES-secp256k1-AES-128-GCM.

  • Key Derivation:

    • Keys are large secret numbers derived from passwords, passphrases, or numbers.
    • Derivation is done using key derivation algorithms.
    • Examples: PBKDF2, Scrypt.

Slide 2.2 : Digital Signatures and Message Authentication

  • Digital Signatures:

    • Guarantees message authenticity, integrity, and non-repudiation.
    • Uses asymmetric key pair (private and public keys).
    • Message is signed with private key and verified with public key.
    • Used in bank systems for signing and approving payments.
    • In blockchain, signed transactions transfer assets between addresses.

  • Message Authentication:

    • Uses message authentication algorithms (e.g., HMAC) and message authentication codes (MAC).
    • Ensures message authenticity, integrity, and authorship.
    • Works alongside encryption to enable secure communication.

Slide 2.3 Secure Random Numbers

  • Random Numbers in Cryptography:
    • Cryptography relies on random numbers and entropy (unpredictable randomness).
    • Ensures secure generation of random numbers, often using CSPRNG (Cryptographically Secure Pseudo-Random Number Generators).
    • Secure random numbers are inherently unpredictable.
    • Developers must ensure proper random number generation, as a broken generator compromises system or app security.

Slide 2.4 Key Exchange

  • Key-Exchange Algorithms:

    • Includes algorithms like Diffie-Hellman and ECDH.
    • Used to securely establish encryption keys between two parties.
    • Ensures secure communication by enabling encrypted message transmission.

  • Key Establishment Schemes:

    • Enable secure establishment of encryption keys.
    • Commonly used when a secure connection is created, such as when opening a modern website or connecting to a WiFi network.

Slide 2.5 Cryptographic Hashes and Password Hashing

  • Cryptographic Hash Functions:

    • Examples: SHA-3, BLAKE2.
    • Transform messages into a fixed-length message digest (hash).
    • Hashes are irreversible and almost uniquely identify the original input.
    • In blockchain, hashes are used for generating addresses, transaction IDs, and in various algorithms.
    • In Git, hashes create unique IDs for files and commits.

  • Password Hashing and Key Derivation:

    • Password hashing protects user passwords and encrypted data.
    • Key derivation functions like Scrypt and Argon2 securely generate hashes or keys from passwords.
    • Involves injecting random parameters (salt) and using multiple iterations to slow down password cracking.

Slide 2.6 Cryptographic Libraries

  • Knowledge of Cryptographic Libraries:

    • Developers should be familiar with modern cryptographic libraries for their specific programming language and platform.
    • Libraries are essential, but understanding the underlying crypto concepts is crucial.

  • Risks of Blindly Using Examples:

    • Copying code from the internet or following blog examples can lead to insecure designs and weak security.

  • Best Practices:

    • Understand the cryptographic concepts before implementing.
    • Carefully choose and combine appropriate algorithms.
    • Adjust parameters to ensure strong security.

Slide 3.6 Public Key Encryption / Decryption

  • Asymmetric Encryption Schemes:

    • Use a pair of cryptographically related public and private keys.
    • Public key is used to encrypt the data.
    • Private key is used to decrypt the encrypted data back to its original form.

  • Key Functionality:

    • Data encrypted with a public key can only be decrypted by the corresponding private key.

  • Ciphertext:
    • The result of encryption is called ciphertext.
    • It is a binary sequence that is unreadable by humans.
    • By design, it cannot be decrypted without the corresponding decryption key.

  • Limitations of Asymmetric Encryption:

    • Asymmetric encryption can only encrypt messages of limited length and is typically slower than symmetric ciphers.

  • Hybrid Encryption (Public-Key Encryption Scheme):

    • Combines symmetric and asymmetric encryption for efficient encryption of larger messages (e.g., PDF documents).
    • Process:
      1. A random symmetric key (sk) is generated.
      2. The message is encrypted symmetrically with sk.
      3. The symmetric key (sk) is then encrypted asymmetrically using the recipient’s public key.
      4. For decryption, the recipient uses their private key to decrypt sk and then uses sk to symmetrically decrypt the message.

  • Key Encapsulation Mechanism (KEM):

    • The process described above is called key encapsulation mechanism (KEM).
    • In KEM, an asymmetrically encrypted random symmetric key is used to securely encrypt data.

  • Alternative Public-Key Encryption:

    • Public-key encryption can also work in the opposite way:
      • Encrypt data with a private key and decrypt it with the public key.
      • This is often used to prove ownership of a private key, while revealing only the public key, as in some digital signature schemes.

Slide 3.7 Signatures: Asymmetric Signing / Verification

  • Digital Signatures:
    • A message signed with a private key generates a digital signature.
    • This signature can later be verified using the corresponding public key.
    • Ensures authenticity, integrity, and non-repudiation of the message.

Slide 4 : Digital Signatures Overview

  • A message is signed using a private key, generating a digital signature.
  • The signature is verified using the corresponding public key.
  • A signed message cannot be altered after signing.
  • Provides authentication, integrity, and non-repudiation (proof of authorship).

Slide 4.1 : Applications of Digital Signatures

  • Finance Industry: Used to authorize payments securely.
  • Operating Systems: OS components and device drivers are digitally signed to prevent malicious code, trojans, or viruses.
  • Blockchain Systems:
    • Transactions are signed by the owner of a blockchain address (linked to a public-private key pair).
    • Proof of authorship: Ensures the transaction was created by the rightful owner and not modified after signing.
    • Essential for digital payments and secure document signing (e.g., contracts).

The RSA Cryptosystem

RSA Public-Key Cryptosystem

  • Based on modular exponentiation (raising numbers to a power by modulus).
  • Security relies on the integer factorization problem (difficulty of factoring large numbers).
  • Commonly used for encryption, digital signatures, and key exchange.
  • We will discuss its details and examples later.

The ECC Cryptosystem

The elliptic-curve cryptography (ECC) public-key cryptosystem is based on the math of the on the algebraic structure of the elliptic curves over finite fields and the difficulty of the elliptic curve discrete logarithm problem (ECDLP). The ECC usually comes together with the ECDSA algorithm (elliptic-curve digital signature algorithm). We shall discuss ECC and ECDSA in details, along with examples.

ECC is Recommended in the General Case

ECC uses smaller keys, ciphertexts and signatures than RSA and is recommended for most applications. It is mathematically proven that a 3072-bit RSA key has similar cryptographic strength to a 256-bit ECC key. Key generation is in ECC is significantly faster than with RSA.

Due to the above reasons most blockchain networks (like Bitcoin and Ethereum) use elliptic-curve-based cryptography (ECC) to secure the transactions.

Note that both RSA and ECC cryptosystems are not quantum-safe, which means that if someone has enough powerful quantum computer, he will be able to derive the private key from given public key in just few seconds.

Slide 9 : Asymmetric Encryption in Practice

Asymmetric encryption works for small messages only (limited by the public / private key length). To encrypt larger messages key encapsulation mechanisms or other techniques can be used, which encrypt asymmetrically a random secret key, then use it to symmetrically encrypt the larger messages. In practice, modern asymmetric encryption schemes involve using a symmetric encryption algorithm together with a public-key cryptosystem, key encapsulation and message authentication.

Popular asymmetric encryption schemes are: RSA-OAEP (based on RSA and OAEP padding), RSAES-PKCS1-v1_5 (based on RSA and PKCS#1 v1.5 padding), DLIES (based on discrete logarithms and symmetric encryption) and ECIES (based on elliptic curve cryptography and symmetric encryption).

Slide 10 : Asymmetric Cryptography Algorithms Explained

Asymmetric cryptography relies on different mathematical techniques to generate secure key pairs. Below are the most commonly used algorithms:

Slide 10.1. RSA (Rivest-Shamir-Adleman)

Overview

RSA is one of the most widely used asymmetric encryption algorithms, named after its creators: Ron Rivest, Adi Shamir, and Leonard Adleman. It is based on the mathematical challenge of prime factorization.

How It Works

Key Generation:

  • Choose two large prime numbers, pp and qq.
  • Compute their product: n=p×qn = p \times q (this forms the modulus).
  • Compute ϕ(n)=(p−1)×(q−1)\phi(n) = (p-1) \times (q-1).
  • Select a public exponent ee such that 1<e<ϕ(n)1 < e < \phi(n) and ee is co-prime to ϕ(n)\phi(n).
  • Compute the private exponent dd, where d≡e−1mod  ϕ(n)d \equiv e^{-1} \mod \phi(n).

Encryption:

  • Ciphertext = C=Memod  nC = M^e \mod n (where MM is the plaintext message).

Decryption:

  • Plaintext = M=Cdmod  nM = C^d \mod n.

Pros:

✔ Highly secure for sufficiently large keys (2048-bit+ recommended).
✔ Used in SSL/TLS, digital signatures, and key exchange.

Cons:

✖ Computationally expensive and slower than symmetric algorithms.
✖ Large key sizes required for strong security.

Slide 10.2 : ECC (Elliptic Curve Cryptography)

Overview

ECC is an alternative to RSA that provides strong security with smaller key sizes. It is based on the difficulty of solving the Elliptic Curve Discrete Logarithm Problem (ECDLP).

How It Works

Uses mathematical operations on points over an elliptic curve defined by an equation like:

y2=x3+ax+by^2 = x^3 + ax + b

Generates public-private key pairs using point multiplication on the curve.

The security comes from the difficulty of solving the inverse operation (discrete logarithm problem).

Pros:

✔ More efficient than RSA – provides equivalent security with smaller key sizes (256-bit ECC ≈ 3072-bit RSA).
✔ Faster encryption and decryption processes.
✔ Ideal for mobile devices due to lower computational requirements.

Cons:

✖ More complex to implement than RSA.
✖ Some concerns about its security against quantum attacks.

** Used In:** Bitcoin, TLS, secure messaging apps (Signal, WhatsApp).

Slide 10.3: Diffie-Hellman (DH) Key Exchange

Overview

Diffie-Hellman is not an encryption algorithm, but a key exchange protocol that enables two parties to securely share a symmetric encryption key over an insecure channel.

How It Works

Two parties (Alice & Bob) agree on a public prime number pp and a base gg.

Each party selects a private secret number:

  • Alice: aa, Bob: bb.
    They compute their public keys:
  • Alice: A=gamod  pA = g^a \mod p
  • Bob: B=gbmod  pB = g^b \mod p
    They exchange public keys and compute the shared secret:
  • Alice: S=Bamod  pS = B^a \mod p
  • Bob: S=Abmod  pS = A^b \mod p
  • Since S=gabmod  pS = g^{ab} \mod p, both parties derive the same key!

Pros:

✔ Provides a secure way to exchange encryption keys over an insecure network.
✔ Used in TLS (HTTPS), VPNs, and SSH.

Cons:

✖ Vulnerable to man-in-the-middle (MITM) attacks if not combined with authentication.
✖ Does not provide encryption or authentication on its own.

Slide 10.4 DSA (Digital Signature Algorithm)

📌 Overview

DSA is specifically designed for digital signatures, providing authentication and integrity verification. Unlike RSA, it cannot be used for encryption.

How It Works

Key Generation:

  • Choose a large prime pp and a smaller prime qq.
  • Select a random private key xx and compute the public key: y=gxmod  py = g^x \mod p.

Signing Process:

  • Generate a random integer kk.
  • Compute the signature components rr and ss: r=(gkmod  p)mod  qr = (g^k \mod p) \mod q s=(k−1(H(m)+xr))mod  qs = (k^{-1} (H(m) + xr)) \mod q
  • The signature is (r, s).

Verification:

  • The receiver checks if the signature is valid using public key yy and verifies integrity.

✅ Pros:

✔ Provides strong authentication and data integrity.
✔ Faster signature generation than RSA.

❌ Cons:

✖ Verification process is slower than RSA.
✖ Requires a secure random number generator for kk (weakness in randomness can break security).

Used In: Government security (FIPS 186-4), Bitcoin transactions.

Slide 11 : Symmetric Cryptography Algorithms Explained

Advanced Encryption Standard (AES) - Rijndael

Introduction

AES (Advanced Encryption Standard), also known as Rijndael, is the most popular symmetric encryption algorithm in modern IT. It is widely used due to its security, speed, and platform support. AES is a 128-bit block cipher that supports 128, 192, or 256-bit secret keys. It is commonly used in block modes like AES-CTR and AES-GCM, which require a random 128-bit initial vector (IV, nonce).

History of AES

  • The AES competition was organized by NIST (1997-2000).
  • Rijndael was selected as the winner and became the official AES encryption standard.
  • In 2001, AES was officially adopted by the US government.
  • No significant weaknesses have been found since its adoption.
  • Rijndael (AES) is free for any use, whether public, private, commercial, or non-commercial.

AES Encryption Process

AES operates using a series of transformations on the input data, divided into fixed-size blocks of 128 bits. The encryption process involves:

  1. Key Expansion: The encryption key is expanded into multiple subkeys for each round.
  2. Initial Round:
    • AddRoundKey: XOR the input block with the first subkey.
  3. Main Rounds (9, 11, or 13 depending on key size):
    • SubBytes: Substitute bytes using a fixed S-box.
    • ShiftRows: Shift rows to add diffusion.
    • MixColumns: Mix data across columns for security.
    • AddRoundKey: XOR with the next subkey.
  4. Final Round:
    • Same as the main rounds, but without MixColumns.

AES Block Cipher Modes

AES is typically used in a block cipher mode to process data securely:

Electronic Codebook (ECB) Mode

  • How It Works: Each plaintext block is encrypted independently with the same key.
  • Security Issue: Identical plaintext blocks result in identical ciphertext blocks, making patterns visible.
  • Use Case: Should never be used for secure encryption, but may be useful for encrypting fixed values like database keys.

Cipher Block Chaining (CBC) Mode

  • How It Works: Each plaintext block is XORed with the previous ciphertext block before encryption.
  • Security Enhancement: Requires an Initialization Vector (IV) to introduce randomness.
  • Weaknesses: Vulnerable to padding oracle attacks if not properly implemented.
  • Use Case: Previously common in SSL/TLS, but replaced by authenticated modes like GCM.

Counter (CTR) Mode

  • How It Works: Converts AES into a stream cipher by encrypting an incrementing counter value.
  • Security Enhancement: Requires a unique nonce for security.
  • Efficiency: Highly parallelizable, making it faster than CBC.
  • Use Case: Used in high-performance encryption applications like disk encryption.

Galois/Counter Mode (GCM)

  • How It Works: Combines CTR mode with Galois Message Authentication Code (GMAC).
  • Security Enhancement: Provides both encryption and integrity verification.
  • Performance: Efficient and supports parallel processing.
  • Use Case: Used in TLS 1.2/1.3, VPNs, and secure communications.

AES Encrypt / Decrypt - Examples

To demonstrate AES encryption and decryption, we will use Python.

Install Required Libraries

pip install pyaes pbkdf2

Simple AES-CTR Example

This example demonstrates AES-256-CTR encryption and decryption without authentication.

Password to Key Derivation

import pyaes, pbkdf2, binascii, os, secrets

# Derive a 256-bit AES encryption key from the password
password = "s3cr3t*c0d3"
passwordSalt = os.urandom(16)
key = pbkdf2.PBKDF2(password, passwordSalt).read(32)
print('AES encryption key:', binascii.hexlify(key))

Explanation:

  • Importing Required Libraries:

    • pyaes: A lightweight AES implementation in pure Python.
    • pbkdf2: Implements PBKDF2 for secure key derivation.
    • binascii: Used for binary-to-ASCII conversions.
    • os: Generates cryptographic random values.
    • secrets: Provides secure random number generation.

  • Password-Based Key Derivation:

    • Uses a password to generate an encryption key.
    • A random 16-byte salt (passwordSalt) is generated to add randomness.
    • PBKDF2 derives a 256-bit encryption key.
    • The key is printed in hexadecimal format.

AES Encryption (CTR Block Mode)

iv = secrets.randbits(256)
plaintext = "Text for encryption"
aes = pyaes.AESModeOfOperationCTR(key, pyaes.Counter(iv))
ciphertext = aes.encrypt(plaintext)
print('Encrypted:', binascii.hexlify(ciphertext))

AES Decryption (CTR Block Mode)

aes = pyaes.AESModeOfOperationCTR(key, pyaes.Counter(iv))
decrypted = aes.decrypt(ciphertext)
print('Decrypted:', decrypted)

Explanation:

  • IV (Nonce) Generation:

    • A 256-bit random IV is generated using secrets.randbits(256).

  • AES-CTR Mode Setup:

    • Initializes AES with CTR mode using the key and IV.
    • CTR mode turns AES into a stream cipher, making encryption faster.

  • Encryption Process:

    • Encrypts the plaintext using AES-CTR mode.
    • The result is stored in ciphertext and printed in hex format.

  • Decryption Process:

    • Uses the same key and IV to decrypt the ciphertext.
    • Retrieves the original plaintext.

AES-GCM provides authenticated encryption and is widely used in secure applications.

AES-256-GCM Encryption & Decryption

from Crypto.Cipher import AES
import binascii, os

def encrypt_AES_GCM(msg, secretKey):
    aesCipher = AES.new(secretKey, AES.MODE_GCM)
    ciphertext, authTag = aesCipher.encrypt_and_digest(msg)
    return (ciphertext, aesCipher.nonce, authTag)

def decrypt_AES_GCM(encryptedMsg, secretKey):
    (ciphertext, nonce, authTag) = encryptedMsg
    aesCipher = AES.new(secretKey, AES.MODE_GCM, nonce)
    plaintext = aesCipher.decrypt_and_verify(ciphertext, authTag)
    return plaintext

secretKey = os.urandom(32)  # 256-bit random encryption key
msg = b'Message for AES-256-GCM'
encryptedMsg = encrypt_AES_GCM(msg, secretKey)
decryptedMsg = decrypt_AES_GCM(encryptedMsg, secretKey)
print("Decrypted message:", decryptedMsg)
